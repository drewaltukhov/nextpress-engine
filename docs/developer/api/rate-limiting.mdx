## How it works

NextPress uses a **token-bucket** algorithm for rate limiting. Each API token gets its own bucket:

- **Default capacity:** 60 requests per minute
- **Burst allowance:** 10 extra requests (handle short spikes)
- **Refill:** continuous, at `capacity / 60` tokens per second

## Per-token overrides

Each token can have a custom `rate_limit_per_minute` value set at creation time. If not set, the site-wide default (60 req/min) applies.

## Response headers

Every API response includes rate-limit headers:

```http
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1714520400
```

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Token's capacity (requests per minute) |
| `X-RateLimit-Remaining` | Tokens left in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the bucket fully refills |

## When you're rate-limited

If you exceed the limit, you receive:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 3
```

```json
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Please retry after 3 seconds."
  }
}
```

Wait for the `Retry-After` duration before retrying.

## Best practices

- **Use reasonable polling intervals** — don't poll faster than once per second
- **Respect `Retry-After`** — implement exponential backoff on 429 responses
- **Request higher limits** if you need them — set `rate_limit_per_minute` at token creation
- **Cache responses** client-side when possible to reduce API calls
